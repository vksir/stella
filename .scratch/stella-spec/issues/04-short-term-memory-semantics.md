# 短期记忆“100K 丢弃”的精确语义

Type: grilling
Status: closed (2026-07-19)
Blocked by: 03

## Question

简报定义短期记忆 = 会话历史，到达 100K 时"自动丢弃超出的部分"。需要拍板精确语义：

- 硬截断（丢最旧消息，可能切断工具调用对）还是接受 SDK 原生 compaction（总结式，旧消息变摘要）？
- "100K"如何计数（token？什么口径）？是整条会话历史的硬上限，还是触发线 + 裁剪目标？
- 倾听模式下被动消息会挤占窗口，裁剪时是否区别对待被动消息与正式对话轮？

依赖 [PI SDK 深度行为事实] 中关于 compaction 与裁剪挂载点的事实。

## Resolution

1. **硬截断，不用 SDK 总结式 compaction**：`compaction.enabled=false`，经 `context` 事件在每次 LLM 调用前沿 user turn 边界丢最旧消息；会话文件保留全量历史，仅 LLM 视野被截断。理由：行为可预测、零额外 token 成本、全量历史留给未来演进；被否备选 = SDK 原生 compaction（摘要质量不可控、每次触发多一次 LLM 调用）与伪 compaction（语义 hack）。
2. **计数口径与语义**：口径直接用 SDK `getContextUsage()`（usage 优先、chars/4 兜底，偏保守方向安全），只计会话历史消息（系统提示词与当轮注入不占额）；上限 = 送入 LLM 的会话历史硬上限，每次 `context` 事件幂等重裁，不设独立裁剪目标水位。
2a. **（2026-07-19 修订）上限值与 TTL**：token 上限为配置项（`shortTermMemory.maxTokens`），**默认 10K**。定 10K 的硬约束：默认值必须 > 最大单轮体量（含两三次工具结果的轮轻松 5~10K；曾提 1K 被否——会被单轮工具调用击穿，裁剪只能沿 user 轮边界丢旧轮、当前轮不能动）。另加**消息年龄上限 3 天**：年龄 >3 天的消息同样淘汰，与 token 上限构成双重淘汰，同一 `context` 事件挂载点执行，仍只裁视野不动会话文件，公私会话同规则。
3. **裁剪一视同仁**：被动倾听消息与正式对话轮纯按时间序同权裁剪，不打标、不做写入侧限流。理由：旧上文价值随时间一致衰减；差别裁剪会让模型看到有时间洞的上下文且算法复杂化；窗口被闲聊淹没的演进路径 = 检索式召回（已录 out of scope）。
